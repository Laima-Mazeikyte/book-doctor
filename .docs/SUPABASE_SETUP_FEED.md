# Supabase setup for cold-start recommendation API

This doc covers:

- **Explicit recommendation flows** (no bandit): `POST /recommend` and `POST /webhook/recommend`
- **Feed/scroll flow** (bandit on): website inserts into `feed_requests` → Supabase webhook → `POST /webhook/feed-recommend` → backend writes to `feed_items` and `feed_log`

`user_cluster_stats` is maintained **only** by the feed webhook handler; explicit recommendation flows do not update it.

---

## 1. API credentials

In **Project Settings → API** copy:

- **Project URL** → `SUPABASE_URL` in `.env`
- **anon public** key → `SUPABASE_ANON_KEY` in `.env`

Optional: `SUPABASE_WEBHOOK_SECRET` in `.env` and set the same value in the Supabase webhook config so the backend can verify webhook requests.

## 2. Tables

### 2.1 `user_ratings` (for webhook flow)

Stores each user’s book ratings. The frontend writes here; the webhook handler reads from here.


| Column        | Type        | Nullable | Notes                                                                         |
| ------------- | ----------- | -------- | ----------------------------------------------------------------------------- |
| `id`          | int8        | no       | Primary key, identity (additional note: this one does not exist int he table) |
| `user_id`     | uuid        | no       | Supabase Auth user id                                                         |
| `book_id`     | text        | no       | Book identifier                                                               |
| `book_rating` | float8      | no       | e.g. 1.0–5.0                                                                  |
| `created_at`  | timestamptz | yes      | Optional, default `now()`                                                     |


**RLS:** Allow `authenticated` (and/or `anon` if your app uses it) to **INSERT**, **SELECT**, **UPDATE**, **DELETE** as needed for your app.

---

### 2.2 `recommendation_requests` (webhook trigger)

One row per “user asked for recommendations”. Frontend inserts a row; Supabase fires a Database Webhook to your backend.


| Column       | Type        | Nullable | Notes                     |
| ------------ | ----------- | -------- | ------------------------- |
| `id`         | int8        | no       | Primary key, identity     |
| `user_id`    | uuid        | no       | Who requested             |
| `created_at` | timestamptz | yes      | Optional, default `now()` |


**RLS:** Allow the role your frontend uses to **INSERT** (and optionally **SELECT**). The backend only needs to be called by the webhook; it doesn’t need direct table access for this table.

---

### 2.3 `recommendation_log` (one row per request)

Written by the backend (direct API and webhook) for logging/analytics.


| Column       | Type        | Nullable | Notes                        |
| ------------ | ----------- | -------- | ---------------------------- |
| `id`         | int8        | no       | Primary key, identity        |
| `request_id` | text        | no       | Links to request (see below) |
| `latency_ms` | float8      | no       | Time to generate (ms)        |
| `user_id`    | uuid        | yes      | User (null for anonymous)    |
| `created_at` | timestamptz | yes      | Optional, default `now()`    |


- **Direct API:** `request_id` is generated by the backend (e.g. timestamp-based).
- **Webhook:** `request_id` = `recommendation_requests.id` (so one log row per recommendation request).

**RLS:** Allow the role your backend uses (e.g. `anon` with service key or a dedicated role) to **INSERT**.

---

### 2.4 `recommendation_items` (one row per recommended book)

Written by the backend with the top‑k book IDs and rank.


| Column       | Type        | Nullable | Notes                           |
| ------------ | ----------- | -------- | ------------------------------- |
| `id`         | int8        | no       | Primary key, identity           |
| `request_id` | text        | no       | Same as in `recommendation_log` |
| `user_id`    | uuid        | yes      | Same as in log                  |
| `book_id`    | text        | no       | Recommended book id             |
| `rank`       | int4        | no       | 1-based rank (1–10)             |
| `created_at` | timestamptz | yes      | Optional, default `now()`       |


**RLS:** Allow the backend’s role to **INSERT**. Frontend typically needs **SELECT** to show results (e.g. filter by `request_id` or `user_id`).

---

### 2.5 `user_cluster_stats` (UCB bandit impressions, feed-only)

Used by the **feed webhook** (`POST /webhook/feed-recommend`) when the UCB bandit is enabled (cluster mapping loaded). One row per (user, cluster): total impressions (how many times that cluster was shown in the feed to that user). The backend reads this to compute UCB and **only the feed webhook** writes/updates it when writing to `feed_items`. Explicit recommendation flows (`/recommend`, `/webhook/recommend`) do **not** update this table, so bandit exploration/exploitation state is not polluted by explicit “recommend” clicks.


| Column        | Type        | Nullable | Notes                                                         |
| ------------- | ----------- | -------- | ------------------------------------------------------------- |
| `user_id`     | uuid        | no       | Supabase Auth user id                                         |
| `cluster_id`  | int4        | no       | Cluster id from bandit mapping                                |
| `impressions` | int4        | no       | Count of recommendations from this cluster shown to this user |
| `created_at`  | timestamptz | yes      | Optional, default `now()`                                     |
| `updated_at`  | timestamptz | yes      | Optional, default `now()`                                     |


- **Unique constraint:** `(user_id, cluster_id)` so the backend can atomically increment impressions on conflict.
- **RLS:** Allow the backend's role to **SELECT**, **INSERT**, **UPDATE** (or **UPSERT** via unique constraint).

Optional env: `SUPABASE_USER_CLUSTER_STATS_TABLE` (default: `user_cluster_stats`).

The server increments feed impressions through a Supabase RPC named `increment_user_cluster_stats` by default. You can rename it with `SUPABASE_USER_CLUSTER_STATS_INCREMENT_RPC`.

### 2.6 Feed flow tables (`feed_requests`, `feed_items`, `feed_log`)

The feed flow uses the same webhook pattern as explicit recommendations: the website writes to Supabase, a Database Webhook calls the backend, and the backend writes results back to Supabase.

`**feed_requests`** (webhook trigger): One row per “user asked for next feed batch”. Frontend inserts a row with `user_id`; Supabase fires a Database Webhook to `POST /webhook/feed-recommend`. The backend updates lifecycle fields so the frontend can observe whether a request is still processing, skipped, failed, or completed.


| Column          | Type        | Nullable | Notes                                                                    |
| --------------- | ----------- | -------- | ------------------------------------------------------------------------ |
| `id`            | int8        | no       | Primary key, identity                                                    |
| `user_id`       | uuid        | no       | Who requested                                                            |
| `status`        | text        | no       | Request state: `pending`, `processing`, `skipped`, `failed`, `completed` |
| `error_message` | text        | yes      | Optional failure/skip reason                                             |
| `result_count`  | int4        | yes      | Number of feed rows written                                              |
| `processed_at`  | timestamptz | yes      | When the backend finished handling the request                           |
| `created_at`    | timestamptz | yes      | Optional, default `now()`                                                |


`**feed_items**` (results): Backend writes ranked feed results per request. The backend also reads prior `feed_items` for the same user to avoid serving duplicate books on later feed requests.


| Column       | Type        | Nullable | Notes                      |
| ------------ | ----------- | -------- | -------------------------- |
| `id`         | int8        | no       | Primary key, identity      |
| `request_id` | text        | no       | Same as `feed_requests.id` |
| `user_id`    | uuid        | yes      | Same as in request         |
| `book_id`    | text        | no       | Recommended book id        |
| `rank`       | int4        | no       | 1-based rank (1–10)        |
| `created_at` | timestamptz | yes      | Optional, default `now()`  |


`**feed_log**` (optional): Backend writes one row per feed request for latency/diagnostics.


| Column       | Type        | Nullable | Notes                     |
| ------------ | ----------- | -------- | ------------------------- |
| `id`         | int8        | no       | Primary key, identity     |
| `request_id` | text        | no       | Links to feed_requests.id |
| `latency_ms` | float8      | no       | Time to generate (ms)     |
| `user_id`    | uuid        | yes      | User                      |
| `created_at` | timestamptz | yes      | Optional, default `now()` |


Install these tables with `[SUPABASE_FEED_TABLES.sql](SUPABASE_FEED_TABLES.sql)`. Configure a **Database Webhook** on `feed_requests` **INSERT** → `https://your-backend-host/webhook/feed-recommend` (see section 4.2).

Optional env: `SUPABASE_FEED_REQUESTS_TABLE`, `SUPABASE_FEED_ITEMS_TABLE`, `SUPABASE_FEED_LOG_TABLE` (defaults: `feed_requests`, `feed_items`, `feed_log`).

## 3. Flows overview


| Flow               | Endpoint / trigger                                           | Bandit | Updates `user_cluster_stats` |
| ------------------ | ------------------------------------------------------------ | ------ | ---------------------------- |
| Explicit (direct)  | `POST /recommend`                                            | No     | No                           |
| Explicit (webhook) | `POST /webhook/recommend`                                    | No     | No                           |
| Feed/scroll        | Insert into `feed_requests` → `POST /webhook/feed-recommend` | Yes    | Yes                          |


- **Explicit:** Full-catalog retrieval; all user ratings used; no cluster restriction.
- **Feed:** Website inserts into `feed_requests`; webhook runs bandit-constrained retrieval, writes `feed_items` and `feed_log`, and atomically increments `user_cluster_stats`. Frontend should watch `feed_requests.status` for lifecycle state and read/subscribe to `feed_items` (e.g. by `request_id` or `user_id`) when the request reaches `completed`.

### 3.1 Feed RPC setup: `increment_user_cluster_stats`

Install the SQL function in `[SUPABASE_INCREMENT_USER_CLUSTER_STATS.sql](c:\\Users\\jkuli\\Desktop\\Books\\script\\SUPABASE_INCREMENT_USER_CLUSTER_STATS.sql)`. The backend calls this RPC to increment impressions atomically, which avoids lost updates when multiple feed requests for the same user happen concurrently.

## 4. Webhook flow (recommendations from Supabase)

1. **Frontend:** User rates books → save rows into `user_ratings` (`user_id`, `book_id`, `book_rating`).
2. **Frontend:** User clicks “Recommend” → insert one row into `recommendation_requests` with `user_id`.
3. **Supabase:** Database Webhook on `recommendation_requests` **INSERT** → POST to your backend, e.g.
  `https://your-api.example.com/webhook/recommend`
4. **Backend:** Receives payload, reads `record.id` (→ `request_id`) and `record.user_id`, queries `user_ratings` for that `user_id`, runs the cold-start model, writes to `recommendation_log` and `recommendation_items`.
5. **Frontend:** Subscribes to `recommendation_items` (e.g. filter by `request_id` or latest by `user_id`) and displays the list.

### 4.1 Configure the Database Webhook in Supabase

1. **Database → Webhooks** (or **Project Settings → Webhooks**).
2. **Create a new webhook:**
  - **Name:** e.g. `on_recommendation_request`
  - **Table:** `recommendation_requests`
  - **Events:** **Insert**
  - **URL:** `https://your-backend-host/webhook/recommend`
  - **HTTP method:** POST  
  - **HTTP headers (optional):**  
  If you set `SUPABASE_WEBHOOK_SECRET` in `.env`, add a header, e.g.  
  `x-webhook-secret: <same secret value>`
3. Save. Supabase will send a JSON body like:

```json
{
  "type": "INSERT",
  "table": "recommendation_requests",
  "schema": "public",
  "record": { "id": 1, "user_id": "uuid-here", "created_at": "..." },
  "old_record": null
}
```

The backend uses `record.id` as `request_id` and `record.user_id` to load `user_ratings` and write `recommendation_items`.

### 4.2 Configure the Feed Database Webhook

1. **Database → Webhooks** (or **Project Settings → Webhooks**).
2. **Create a new webhook:**
  - **Name:** e.g. `on_feed_request`
  - **Table:** `feed_requests`
  - **Events:** **Insert**
  - **URL:** `https://your-backend-host/webhook/feed-recommend`
  - **HTTP method:** POST  
  - **HTTP headers (optional):** If you set `SUPABASE_WEBHOOK_SECRET` in `.env`, add  
  `x-webhook-secret: <same secret value>`
3. Save. Supabase will send a JSON body like:

```json
{
  "type": "INSERT",
  "table": "feed_requests",
  "schema": "public",
  "record": { "id": 1, "user_id": "uuid-here", "created_at": "..." },
  "old_record": null
}
```

The backend uses `record.id` as `request_id` and `record.user_id` to load `user_ratings`, run bandit retrieval, write `feed_items` and `feed_log`, increment `user_cluster_stats`, and update `feed_requests.status`.

### 4.3 Env for webhook

- `SUPABASE_URL`, `SUPABASE_ANON_KEY` (or service role) so the backend can read `user_ratings` and write `recommendation_log` / `recommendation_items`.
- Optional: `SUPABASE_WEBHOOK_SECRET` and the same value in the webhook header so the backend rejects unknown callers.
- Optional: `SUPABASE_USER_RATINGS_TABLE`, `SUPABASE_RECOMMENDATION_REQUESTS_TABLE` if you use different table names (defaults: `user_ratings`, `recommendation_requests`).
- Optional (feed): `SUPABASE_FEED_REQUESTS_TABLE`, `SUPABASE_FEED_ITEMS_TABLE`, `SUPABASE_FEED_LOG_TABLE` (defaults: `feed_requests`, `feed_items`, `feed_log`).
